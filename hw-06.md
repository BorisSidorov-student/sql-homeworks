## Решение 1
Самое ключевое различие в данных подходах – это возможность осуществлять запись и чтение данных на разных серверах.

Допустим, у нас есть сервер, на котором развернута **`СУБД`**, пока без настроенной репликации. Обращаясь к **`БД`** на данном сервере под **`УЗ`**, у которой есть соответствующие привилегии, мы можем производить как чтение данных (различные запросы **`SELECT`**), так и запись данных (запросы **`INSERT`**, **`UPDATE`**, **`DELETE`**, **`ALTER`** – любые запросы, связанные с изменением данных в **`БД`**). Такой сервер у нас один, и он по умолчанию уже является **`master`**.

И вот наступила задача настроить репликацию в целях отказоустойчивости работы **`СУБД`**. Поднимаем новый инстанс **`СУБД`** – он будет выступать в роли **`slave`**. Настраиваем этот инстанс на репликацию данных из основного инстанса, то есть источником данных будет **`master`** **`СУБД`**. Получается, что у нас есть основной инстанс (**`master`**), через который происходит и чтение, и запись данных, и появился ещё один инстанс (**`slave`**), который выступает в роли резерва. На этот инстанс будет происходить копирование всех изменённых данных из **`master`**. Цель инстанса **`slave`** – быть максимально таким же, как **`master`**, при классической настройке репликации, когда репликация настроена на все **`БД`** в **`СУБД`**, все таблицы и все столбцы (полная копия инстанса **`master`**). Это нужно, чтобы **`slave`** в любую секунду был готов принять на себя роль **`master`**. Но до этого момента на **`slave`** должна быть возможность осуществлять только запросы на чтение, иначе **`slave`** перестанет быть копией инстанса **`master`**. Если же в **`slave`** дать возможность вносить изменения в какую-либо **`БД`**, то когда наступит момент переключения в режим **`master`**, данные на **`slave`** будут отличаться от данных инстанса **`master`**, и **`БД`** перестанет адекватно работать – получится рассинхрон данных, и приложения уже не будут работать так, как должны по логике. Это был подход **`master -> slave`**.

В режиме **`master – master`** ключевое отличие будет в том, что наш новый инстанс, который мы развернём (назовём его **`master2`**), будет не только постоянно мониторить изменения в первом инстансе, но и первый инстанс, который работал изначально (назовём его **`master1`**), также будет мониторить изменения нового инстанса **`master2`**. Получается такая картина, что оба инстанса друг друга мониторят и копируют любые изменения, происходящие в них. Таким образом, и в инстанс **`master1`**, и в инстанс **`master2`** можно осуществлять любые запросы – как на чтение, так и на запись данных.

Ранее была произведена настройка сервера репликации классическим способом, основанная на позиции бинарного лога, а также все транзакции были анонимными. Этот режим упрощает добавление новых серверов репликации, где не нужно будет указывать номер бинарного лога и его позицию, а просто в конфигурации реплики добавить пункт **`SOURCE_AUTO_POSITION = 1`**. Почитав документацию, я понял, что это можно сделать, так сказать, “на горячую” уже с настроенной репликацией, использующей анонимные транзакции. Для имитации такого процесса я попросил нейросеть написать небольшую процедуру, которая будет добавлять раз в 5 секунд записи в таблицу **`actor`** **`БД`** **`sakila`**:

    DELIMITER $$

    CREATE PROCEDURE generate_load()
    BEGIN
        DECLARE v_id INT DEFAULT 202;
        DECLARE v_name VARCHAR(45);
        
        -- Бесконечный цикл
        WHILE TRUE DO
            -- Формируем имя с номером для наглядности
            SET v_name = CONCAT('test', v_id);
            
            -- Вставляем новую строку
            INSERT INTO actor (actor_id, first_name, last_name, last_update)
            VALUES (v_id, v_name, 'load_test', NOW());
            
            -- Увеличиваем счётчик
            SET v_id = v_id + 1;
            
            -- Пауза 30 секунд
            DO SLEEP(30);
            
            -- Если достигли 300, удаляем все вставленные строки (202-300)
            IF v_id > 300 THEN
                DELETE FROM actor WHERE actor_id BETWEEN 202 AND 300;
                SET v_id = 202;  -- Сбрасываем счётчик
            END IF;
        END WHILE;
    END$$

    DELIMITER ;

**Групповая репликация MySQL.**

Ранее я реализовывал репликацию данных классическим простым способом, как с анонимными транзакциями, так и более современным подходом через глобальные идентификаторы транзакций (**`GTID`**). Но этот подход хорош для обычной репликации, однако если возникнут какие-либо проблемы с сервером-источником, то придётся вручную осуществлять переключение с реплики на источник. Для решения подобных проблем существует подход использования групповой репликации – как с одним источником, так и с несколькими. Преимущества групповой репликации заключаются в её отказоустойчивости: если выйдет из строя сервер-источник, то группа из нескольких серверов придёт к решению большинства и назначит новый источник без вмешательства администратора. После того как сервер возобновит свою работу, он снова будет добавлен в группу (опять же по согласованию большинства членов групповой репликации) и через специальные внутренние процессы восстановления данных догонит группу и вновь станет её полноправным членом. Такую группу я попробую настроить.

Количество серверов в такой группе должно быть нечётным для формирования большинства (кворума): минимум **3**, максимум **9**. Я создам группу из **3** серверов. Есть множество требований и ограничений для обеспечения надёжной работы такой группы, самый важный – режим **`GTID`**, в который я уже перевёл во втором задании.

Первым делом я начну работу с основного сервера **`DB01-mysql-8.0.45`**. Я пересоздал контейнер с нуля, использовав отредактированный файл конфигурации; к тому же у меня он будет не один.

![ls -l](screen/hw-06/task-3/hw-06-3.15.png)

Основной файл конфигурации **`10-main.cnf`**, он будет использоваться при инициализации **`СУБД`**, и в нём такое наполнение:

    [mysqld] 
    server-id=1
    report_host=DB01-mysql-8.0.45
    gtid_mode=ON
    enforce_gtid_consistency=ON
    log_bin=binlog
    disabled_storage_engines="MyISAM,BLACKHOLE,FEDERATED,ARCHIVE,MEMORY"
    log_replica_updates=ON
    binlog_checksum=CRC32
    lower_case_table_names=1

Все эти данные я добавлял из требований и рекомендаций официальной документации. Прокомментирую самые важные:

- **`report_host=DB01-mysql-8.0.45`** – для корректного разрешения доменных имён, чтобы члены группы могли корректно находить сервер **`СУБД`**, явно указываю имя контейнера (или **`DNS`** имя сервера, или статический **`IP`**) в данную переменную.
- **`disabled_storage_engines=`** – явно отключаю движки хранения данных, с которыми групповая репликация не будет работать.
- **`lower_case_table_names=`** – рекомендация от разработчиков использовать стиль в нижнем регистре, но если уже есть данные, то нужно будет переводить под этот стиль.

Далее – настройки, связанные с включением **`GTID`**, принудительная работа с транзакциями в режиме **`GTID`**, настройки, касающиеся оптимизации работы с бинарными логами.

Во втором конфигурационном файле **`20-gr.cnf.inactive`** содержатся настройки уже непосредственно групповой репликации:

    [mysqld]
    plugin_load_add="group_replication.so;mysql_clone.so"
    group_replication_group_name="c2062b54-41d7-11f1-bda3-baf33d20d4db"
    group_replication_start_on_boot=off
    group_replication_local_address= "DB01-mysql-8.0.45:33061"
    group_replication_group_seeds= "DB01-mysql-8.0.45:33061,DB02-mysql-8.0.45:33061,DB03-mysql-8.0.45:33061"
    group_replication_bootstrap_group=off
    group_replication_recovery_get_public_key = ON

Пояснения:

- **`plugin_load_add=`** – загрузка плагинов для работы групповой репликации и клонирования данных.
- **`group_replication_group_name=`** – уникальное имя группы; можно сгенерировать обычным запросом в **`cli`**-клиенте: **`SELECT UUID()`**.
- **`group_replication_start_on_boot=`** – запуск групповой репликации при старте **`СУБД`**; включаю после того, как сервер (контейнер) будет настроен.
- **`group_replication_local_address=`** – специальный адрес для связи с членами группы по принятию решений о добавлении или исключении сервера из группы, обработки (сертификации) транзакций, то есть адрес для внутренней связи группы, а не подключения клиентов для работы с данными.
- **`group_replication_group_seeds=`** – список всех участников, которые планируются быть в группе; я указал абсолютно все сервера, в том числе и собственный адрес. По этому списку сервер поочерёдно будет пробовать установить соединения для того, чтобы подать заявку на вступление в группу.
- **`group_replication_bootstrap_group=`** – инициализация процесса создания новой групповой репликации; явно указываю **`off`**, так как эта опция включается только единожды при создании группы и только на первичном сервере.
- **`group_replication_recovery_get_public_key =`** – запрос публичного **`RSA`** ключа для шифрования пароля **`УЗ`** для работы групповой репликации по зашифрованному каналу.
