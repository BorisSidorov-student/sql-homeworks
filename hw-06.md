## Решение 1
Самое ключевое различие в данных подходах – это возможность осуществлять запись и чтение данных на разных серверах.

Допустим, у нас есть сервер, на котором развернута **`СУБД`**, пока без настроенной репликации. Обращаясь к **`БД`** на данном сервере под **`УЗ`**, у которой есть соответствующие привилегии, мы можем производить как чтение данных (различные запросы **`SELECT`**), так и запись данных (запросы **`INSERT`**, **`UPDATE`**, **`DELETE`**, **`ALTER`** – любые запросы, связанные с изменением данных в **`БД`**). Такой сервер у нас один, и он по умолчанию уже является **`master`**.

И вот наступила задача настроить репликацию в целях отказоустойчивости работы **`СУБД`**. Поднимаем новый инстанс **`СУБД`** – он будет выступать в роли **`slave`**. Настраиваем этот инстанс на репликацию данных из основного инстанса, то есть источником данных будет **`master`** **`СУБД`**. Получается, что у нас есть основной инстанс (**`master`**), через который происходит и чтение, и запись данных, и появился ещё один инстанс (**`slave`**), который выступает в роли резерва. На этот инстанс будет происходить копирование всех изменённых данных из **`master`**. Цель инстанса **`slave`** – быть максимально таким же, как **`master`**, при классической настройке репликации, когда репликация настроена на все **`БД`** в **`СУБД`**, все таблицы и все столбцы (полная копия инстанса **`master`**). Это нужно, чтобы **`slave`** в любую секунду был готов принять на себя роль **`master`**. Но до этого момента на **`slave`** должна быть возможность осуществлять только запросы на чтение, иначе **`slave`** перестанет быть копией инстанса **`master`**. Если же в **`slave`** дать возможность вносить изменения в какую-либо **`БД`**, то когда наступит момент переключения в режим **`master`**, данные на **`slave`** будут отличаться от данных инстанса **`master`**, и **`БД`** перестанет адекватно работать – получится рассинхрон данных, и приложения уже не будут работать так, как должны по логике. Это был подход **`master -> slave`**.

В режиме **`master – master`** ключевое отличие будет в том, что наш новый инстанс, который мы развернём (назовём его **`master2`**), будет не только постоянно мониторить изменения в первом инстансе, но и первый инстанс, который работал изначально (назовём его **`master1`**), также будет мониторить изменения нового инстанса **`master2`**. Получается такая картина, что оба инстанса друг друга мониторят и копируют любые изменения, происходящие в них. Таким образом, и в инстанс **`master1`**, и в инстанс **`master2`** можно осуществлять любые запросы – как на чтение, так и на запись данных.

Ранее была произведена настройка сервера репликации классическим способом, основанная на позиции бинарного лога, а также все транзакции были анонимными. Этот режим упрощает добавление новых серверов репликации, где не нужно будет указывать номер бинарного лога и его позицию, а просто в конфигурации реплики добавить пункт **`SOURCE_AUTO_POSITION = 1`**. Почитав документацию, я понял, что это можно сделать, так сказать, “на горячую” уже с настроенной репликацией, использующей анонимные транзакции. Для имитации такого процесса я попросил нейросеть написать небольшую процедуру, которая будет добавлять раз в 5 секунд записи в таблицу **`actor`** **`БД`** **`sakila`**:

    DELIMITER $$

    CREATE PROCEDURE generate_load()
    BEGIN
        DECLARE v_id INT DEFAULT 202;
        DECLARE v_name VARCHAR(45);
        
        -- Бесконечный цикл
        WHILE TRUE DO
            -- Формируем имя с номером для наглядности
            SET v_name = CONCAT('test', v_id);
            
            -- Вставляем новую строку
            INSERT INTO actor (actor_id, first_name, last_name, last_update)
            VALUES (v_id, v_name, 'load_test', NOW());
            
            -- Увеличиваем счётчик
            SET v_id = v_id + 1;
            
            -- Пауза 30 секунд
            DO SLEEP(30);
            
            -- Если достигли 300, удаляем все вставленные строки (202-300)
            IF v_id > 300 THEN
                DELETE FROM actor WHERE actor_id BETWEEN 202 AND 300;
                SET v_id = 202;  -- Сбрасываем счётчик
            END IF;
        END WHILE;
    END$$

    DELIMITER ;

3. Выполняю команду:

    SET @@GLOBAL.GTID_MODE = OFF_PERMISSIVE;

Этим я разрешаю принимать помимо анонимных транзакций ещё и **`GTID`**-транзакции, но сам сервер будет генерировать пока анонимные транзакции.

![set OFF_PERMISSIVE](screen/hw-06/task-2/hw-06-2.38.png)

На этом этапе полезно проверить переменную **`GTID_OWNED`**:

    SELECT @@GLOBAL.GTID_OWNED;

![SELECT @@GLOBAL.GTID_OWNED](screen/hw-06/task-2/hw-06-2.29.png)

Если она пуста, то это значит, что если были какие-либо **`GTID`**-транзакции, которые сервер начал принимать, они были обработаны, и можно смело переходить к следующему шагу.

4. Далее выполняю команду:

    SET @@GLOBAL.GTID_MODE = ON_PERMISSIVE;

После ввода этой команды наш сервер полностью перейдёт в режим **`GTID`** и сам начнёт генерировать **`GTID`**-транзакции. В то же время анонимным транзакциям доступ ещё не закрыт – они могут поступать, и сервер будет с ними работать. На этом этапе нужно всё полностью перевести на **`GTID`**, ведь дальше мы будем запрещать анонимные транзакции. Для этого сказано проверять статус переменной:

    SHOW STATUS LIKE 'ONGOING_ANONYMOUS_TRANSACTION_COUNT';

Эта переменная хранит значение – сколько на данный момент есть выполняющихся анонимных транзакций. Нужно дождаться, пока счётчик не станет **`0`**.

![SHOW STATUS](screen/hw-06/task-2/hw-06-2.30.png)

Также с этого момента наш сервер сам уже генерирует **`GTID`**-транзакции, и на реплике мы по идее должны увидеть последнюю из них в статусе реплики.

![show variables](screen/hw-06/task-2/hw-06-2.31.png)

Вижу, что реплика успешно обрабатывает уже **`GTID`**-транзакции.

5. Данный этап – проверочный и финальный перед полным переключением на режим **`GTID`**, где анонимные транзакции уже не будут доступны. В документации сказано, что лучше всего дать поработать в таком режиме, пока все бинарные логи, содержащие записи об анонимных транзакциях, не будут удалены, а также когда старые бекапы, где были анонимные транзакции, больше не понадобятся. Я ускорю этот процесс, так как в продакшене это может длиться несколько месяцев или больше – всё зависит от инфраструктуры и политики компании.

Для того чтобы перевести все записи в новый бинарный лог, существует команда **`FLUSH LOGS`** – после неё создастся новый инкрементный лог, и дальше все записи будут продолжаться именно в него. Посмотрю, какие логи сейчас есть на текущий момент, командой:

    SHOW BINARY LOGS;

![SHOW BINARY LOGS](screen/hw-06/task-2/hw-06-2.32.png)

Вижу, что последний номер – **`000004`**. Теперь выполню:

    FLUSH LOGS;

![FLUSH LOGS](screen/hw-06/task-2/hw-06-2.33.png)

Начиная с лога **`000005`** я уверен, что там есть только **`GTID`**-транзакции. Теперь затру все логи до номера **`000005`**:

    PURGE BINARY LOGS TO 'binlog.000005';

![PURGE BINARY LOGS](screen/hw-06/task-2/hw-06-2.34.png)

То же самое проделаю и на реплике.  
Теперь сделаю полный бекап перед полным переключением на **`GTID`**. Воспользуюсь утилитой **`mysqldump`**, только добавлю в дамп информацию о выполненных **`GTID`**-транзакциях ключом **`--set-gtid-purged=ON`**, а ключик **`--source-data=2`** комментирует координаты бинарного лога (если вдруг они потребуются для новых реплик):

    mysqldump -u root -p --all-databases --single-transaction --source-data=2 --set-gtid-purged=ON > /files/source_data_with_gtid.sql

![mysqldump](screen/hw-06/task-2/hw-06-2.35.png)

6. Финальный этап. Перевожу полностью **`СУБД`** в режим **`GTID`**:

    SET @@GLOBAL.GTID_MODE = ON;

![show variables](screen/hw-06/task-2/hw-06-2.36.png)

С этого момента **`СУБД MySQL`** работает в режиме **`GTID`**.

7. Теперь необходимо отредактировать конфигурационный файл, добавив в него пару строчек как на мастере, так и на сервере:

    gtid_mode=ON
    enforce_gtid_consistency=ON

![cat config](screen/hw-06/task-2/hw-06-2.37.png)

8. Теперь в настройках клиента реплики допишем строку, связанную с автопозиционированием на основе **`GTID`**, предварительно остановив реплику.

![change replication](screen/hw-06/task-2/hw-06-2.38.png)

Готово. **`СУБД`** полностью переведены в режим **`GTID`**, и можно добавлять новый сервер, уже используя автопозиционирование, не заморачиваясь с координатами файлов.

Подниму **3-й** сервер с конфигурационным файлом, в котором он уже будет работать в режиме **`GTID`**.

![docker run](screen/hw-06/task-2/hw-06-2.39.png)

Импортирую бекап, который делал перед полным включением, и настраиваю реплику уже с **`GTID`**.
